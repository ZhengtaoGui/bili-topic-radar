# B站选题撞车雷达 — 设计稿 (v0.2)

> An MCP server + agent skill that helps Bilibili creators de-risk topic
> selection: before you make a video, it researches who's already done it,
> how it went viral, what angle is missing — and proposes differentiated
> topics with a clear 做 / 改角度 / 延后 / 放弃 recommendation.

作者协作:Claude(架构 / agent 流水线 / 策略判断) + Codex(数据采集层 / 检索 / 报告渲染 / 原型工程)。

---

## 1. 定位与痛点

**给谁:** 知识区 / 数码区 / 游戏区的中小 UP 主(几千到几十万粉)。

**痛点:** 选题是创作里风险最高、最靠玄学的一步。常见翻车:

- **撞车** —— 兴冲冲做完,发现同题视频满地都是,且别人播放量碾压。
- **同质化** —— 又做了一期"和所有人一样"的入门科普,没有差异点。
- **时机错位** —— 热点已经凉了才入场,或太早没人搜。
- **盲点** —— 不知道观众在评论区反复求而没人做的那个角度。

**这个 agent 做的事:** 把"选题前的尽调"自动化——不是给你一个热榜,而是给你一份**带证据的竞争态势 + 差异化方案 + 该不该做的判断**。

---

## 2. 为什么需要 agent,而不是一个爬虫脚本

写死的脚本只能"抓 + 排序",做不到下面这些**判断**:

| 环节 | 需要的判断 | 脚本为什么不行 |
|------|-----------|---------------|
| 查询扩展 | 把"讲讲 MCP"扩成 8 个真实搜索词(同义、子主题、相关角度) | 关键词联想需要语义理解 |
| 角度聚类 | 把已有 40 个视频归成"入门科普/实战教程/踩坑/对比评测/深度原理"几类 | 同一主题不同切法,靠标题字面分不开 |
| 同质化打分 | 判断每个角度"饱和还是空缺",结合数量+时效+播放分布 | 需要综合判断,不是单一指标 |
| 缺口挖掘 | 从高赞评论里读出"观众反复求但没人做"的诉求 | 需要从自然语言里抽意图 |
| 时机判断 | 这个题在"上升/平稳/衰退",值不值得蹭 | 需要结合发布时间×播放趋势做推断 |
| 最终决策 | 做 / 改角度 / 延后 / 放弃,并说清理由 | 这是策略判断,脚本只能报数字 |

**一句话:** 数据采集是工具(MCP 干),选题策略是判断(agent 干)。这正是 agent 项目的价值分界。

## 2.5 先看一个通俗例子(完整跑一遍)

> 老王,数码区 UP 主,2 万粉,想做一期讲「MCP」的视频。他打开雷达,输入 "MCP",几分钟后看到:
>
> 1. **这题现在火不火**(像大众点评看"这家店多火"):
>    "全站 47 个相关视频,近 3 个月新冒出 12 个 —— 正在升温,不冷门也还没烂大街。"
> 2. **哪个角度还有空位**(像超市货架,看哪格满了哪格空):
>    "『5 分钟入门』这格已经被 9 个视频塞满,头部 32 万播放,你挤不进去;但
>    『踩坑避雷』和『跟 Function Calling 到底差在哪』这两格**是空的**。"
> 3. **观众在评论区喊什么**(直接把热评原文念给你听):
>    "点赞 1200 的热评写着『原理讲太多了,求一个真能跑起来的端到端 demo』
>    —— 这就是没人满足的需求。"
> 4. **给你三个差异化方案**:其中"方案A:30 分钟把你的系统接进 Claude(实战向)",
>    雷达说"这个角度现在 0 竞争,且正好踩中那条 1200 赞的诉求"。
> 5. **一句话拍板**:"别做入门科普(已经饱和你打不过),做『实战落地』,
>    窗口还有 2-3 周,要做趁早。"
>
> 老王拿到的不是一张冷冰冰的热榜,而是一个**懂行的朋友**帮他把"做不做、做哪个、
> 为什么、什么时候做"全想明白了。这就是 agent 和爬虫的区别。

## 3. 系统架构

沿用 Gradescope 项目验证过的好模式:**MCP server 提供只读数据工具 + 一个 skill 固化策略流程**。

```
┌─────────────────────────────────────────────────────────────┐
│  Agent (Claude) + skill: bilibili-topic-radar                │
│  ── 编排多步流水线、做所有判断、人在回路                       │
└───────────────┬─────────────────────────────────────────────┘
                │ 调用 MCP 工具(只读)
┌───────────────▼─────────────────────────────────────────────┐
│  MCP Server: bilibili-topic-radar                            │
│  tool_search_topic / tool_partition_trending /               │
│  tool_video_stats / tool_video_hot_comments /                │
│  tool_video_subtitle(可选) / tool_cross_search(可选)        │
└───────────────┬─────────────────────────────────────────────┘
                │
┌───────────────▼─────────────────────────────────────────────┐
│  Data Layer (Codex 主写): bilibili-api-python + httpx        │
│  WBI 签名 · 限流 · 缓存 · 重试 · 数据 schema                  │
└──────────────────────────────────────────────────────────────┘
```

**为什么是只读:** 整个项目不碰任何账号写操作(不发评论、不发动态)。零封号风险、零 ToS 灰区,采集的全是公开数据。

## 4. Agent 流水线(判断逻辑住在这里)

输入:UP 主的一句选题想法(可带分区、粉丝量、频道风格)。
例:「我想做一期讲 MCP(Model Context Protocol)的视频,数码/科技区,粉丝 2 万」

```
Step 1  查询扩展      想法 → 6-10 个真实搜索词(同义/子题/相关角度)
Step 2  采集(Codex)  对每个词搜 top-N 视频 + 拉最相关几条的高赞评论
Step 3  去重聚类      把已有视频按"角度"归类,生成角度地图
Step 4  同质化打分    每个角度算饱和度(数量×时效×播放分布×互动率)
Step 5  缺口挖掘      从高赞评论里抽"未被满足的诉求"(带原文引用)
Step 6  时机判断      该题在上升/平稳/衰退;能不能蹭、要不要赶紧做
Step 7  方案生成      产出 3 个差异化选题(角度+证据+标题候选+竞争度+风险)
Step 8  人在回路      出报告 → UP 主选 / 让 agent 再细化某一个
```

**关键判断点的设计原则:**

- **Step 4 饱和度** 不是单看视频数量。一个角度有 50 个视频但全是半年前的旧坑 + 播放分散 → 其实是"看似拥挤实则有空间"。打分要综合:近 90 天视频占比、头部视频是否垄断播放(基尼系数感觉)、平均互动率。
- **Step 5 缺口挖掘** 是整个项目的"金矿"。高赞评论里的「讲得太浅了求进阶」「没讲到 X」「能不能出个实战」就是差异化的直接依据。每个挖出的缺口必须**附带评论原文 + 点赞数**作为证据,不能让 agent 凭空说"观众想要 X"。
- **Step 7 方案** 每个选题必须可证伪:写清"它差异在哪、依据是 Step5 的哪条缺口/Step4 的哪个空缺角度",而不是拍脑袋。
- **总体建议** 允许而且鼓励说"**放弃 / 延后**"。如果某题已被头部彻底占领且无缺口,agent 的诚实价值就是劝退,而不是硬凑三个方案。

## 5. 数据采集层(B站接口现实方案)

> 基于公开端点 + `bilibili-api-python` 的现状。标 ⚠️ 的项需在原型阶段**上线实测确认**。

| 能力 | 端点 / 库方法 | 登录? | WBI 签名? | 关键返回字段 |
|------|--------------|-------|-----------|-------------|
| 关键词搜视频 | `x/web-interface/wbi/search/type` · `bilibili_api.search` | 否(匿名可搜) | **是**(库已处理) | bvid, title, description, play(播放), video_review(弹幕), pubdate, duration, author, mid, tag |
| 分区排行榜 | `x/web-interface/ranking/v2?rid={tid}` · `bilibili_api.rank` | 否 | 否 | 同上 + 排名 |
| 视频统计/元数据 | `x/web-interface/view?bvid=` · `video.Video.get_info()` | 否 | 部分 | stat{view,danmaku,reply,like,coin,favorite,share}, pubdate, owner |
| 高赞评论 | `x/v2/reply/wbi/main?mode=3`(热门排序) · `bilibili_api.comment` | 否 | **是** ⚠️ | message, like, ctime, member.uname |
| 视频字幕(可选) | `x/player/wbi/v2` → subtitle_url | **多数需 SESSDATA** ⚠️ | 是 | subtitle.subtitles[].subtitle_url(json) |

**WBI 签名:** 2023 年后搜索/评论等接口要求 WBI 签名(先从 `x/web-interface/nav` 取 img_key/sub_key 再做 md5 mixin)。`bilibili-api-python` 已内置处理 ——⚠️ 仍建议原型阶段实跑一次确认库版本 OK。

**反爬 / 限流(给具体数字):**
- 请求间隔 **1.5–2s + 随机抖动**,并发 **1**(单线程顺序)。撞 `-412`(风控)就指数退避。
- 必带真实 `User-Agent` + `Referer: https://www.bilibili.com`。
- 可选配 `SESSDATA` cookie 提升部分接口稳定性 / 解锁字幕;**纯搜索+评论匿名通常够用**。
- 全程**本地缓存**(搜索结果/视频信息/评论按 key 落盘,TTL 几小时),避免重复打点、也利于离线 demo。

**坑与不稳定点(Codex 实现时注意):**
- 搜索结果里 `description`、`tag` 字段偶尔缺失或被 HTML 转义,要兜底。
- `play`/`video_review` 在搜索结果是字符串("1.2万"风格)还是数字,版本不一,需归一化。
- 评论接口分页游标在改版,`mode=3` 热门排序偶有变动 ⚠️ 实测。
- 字幕**很多视频根本没有**(尤其没开 AI 字幕的),把它设计成"有就用、没有就跳过",不作为主链路。
- **隐私/ToS:** 只读公开数据;评论挖掘只做聚合 + 必要引用,不收集 / 不展示评论者画像;不缓存超出分析所需的个人信息。

## 6. MCP 工具清单(server 暴露给 agent 的接口)

```
tool_search_topic(keyword, partition=None, limit=30, order="totalrank")
    → 按词搜视频,返回 VideoCard 列表(供 Step2)

tool_partition_trending(partition, period="weekly")
    → 分区排行榜/热门(给"大盘热度"参照)

tool_get_video_stats(bvid)
    → 单视频精确统计与元数据

tool_get_video_hot_comments(bvid, limit=20)
    → 高赞热门评论(供 Step5 缺口挖掘)

tool_get_video_subtitle(bvid)           # 可选,有就返回,没有就说明
    → 字幕文本(供深度内容对比)

tool_cross_search(query, sources=["zhihu","weibo"])   # 可选,二期
    → 跨平台热度参照(可用 WebSearch 兜底)
```

每个工具支持 `output_format="json"`(给 agent 规划用)/ `"markdown"`(给人看)。

## 7. 输出:选题雷达报告(成品形态)

一份 markdown 报告 + 一个 JSON(供后续程序消费)。报告骨架:

```markdown

## 一、竞争态势
- 命中 47 个相关视频,近 90 天 12 个(↑ 上升期)
- 头部 3 个视频吃掉 68% 播放(头部垄断,但腰部有空间)

| 标题 | UP主 | 播放 | 发布 | 角度 |
|------|------|------|------|------|
| 5分钟看懂MCP | … | 32万 | 18天前 | 入门科普 |
| …

## 二、角度地图(已被占领 vs 空缺)
- 🔴 入门科普(饱和,9 个,头部强)
- 🟡 实战教程(半饱和,4 个,但都偏旧)
- 🟢 踩坑/避雷(空缺,0 个)
- 🟢 和 Function Calling 对比(空缺)

## 三、观众未被满足的需求(证据来自高赞评论)
1. 「讲原理的太多,求一个真能跑起来的端到端 demo」—— 👍 1.2k
2. 「能不能讲讲企业内部怎么落地、安全怎么做」—— 👍 430
3. …

## 四、时机判断
上升期,近两周搜索/出片量明显增加,建议 **2 周内** 入场。

## 五、3 个差异化选题方案
### 方案A:「MCP 实战:30 分钟把你的内部系统接进 Claude」
- 差异点:填补"端到端实战"空缺(依据 角度地图🟢 + 评论#1)
- 标题候选 / 封面方向 / 预计竞争度:低 / 风险:对剪辑要求高
### 方案B … ### 方案C …

## 六、总体建议
✅ 做,但**别做入门科普(已饱和)**,走"实战落地"角度。窗口期约 2-3 周。
```

> 注意:报告里所有"观众想要 X"必须挂证据(评论原文+赞数);标题候选**只给方向,不教标题党**,不承诺播放量。

## 7.5 进阶能力(从"查重工具"升级成"选题智能体")

v0 是"帮你查这题撞不撞车"。进阶版要回答更难的问题:**"我接下来该做什么、什么时候做、为什么我能赢。"** 七个进阶模块,大多复用同一套数据层:

**A1. 选题生命周期建模 + 时机预测**
- 把一个题的所有视频按"发布时间 × 播放速度"画成曲线,判断它处于:萌芽 / 上升 / 爆发 / 饱和 / 衰退,并预测"窗口期还剩多久"。
- 通俗:像看股票 K 线 —— 告诉你这只"题"是刚启动,还是已经在高位接盘。

**A2. 爆款基因拆解**
- 对该题 top 5 爆款,拆"它们凭什么火":标题套路、封面风格、开头 15 秒钩子(从字幕 + 弹幕第一个高峰提取)、节奏。提炼成可复用模式。
- 通俗:不是教你抄,而是告诉你"这条赛道的观众吃哪一套",比如"开头 10 秒不抛结论就掉粉"。

**A3. 机会分(Opportunity Score)**
- 给每个候选选题打 0–100 分:`缺口大小 × 热度趋势 × (1 − 竞争度) × 与你频道匹配度`。
- 通俗:像点评里的"性价比之王"标签,把一堆模糊感觉压成一个能横向比的分。

**A4. 个性化匹配(你到底打不打得过)**
- 结合**你自己频道**的历史爆款 / 风格 / 粉丝量,只推你"吃得动"的题。
- 通俗:同一个缺口,头部大 UP 能吃,你 2 万粉吃不动;雷达会避开"看得到吃不到"的题。

**A5. 跨平台先行指标(预判,而不是追热点)**
- 把知乎热榜 / 微博 / 小红书 / Google Trends / GitHub trending 当"先行信号",抓"别处已经热、B站还没人做"的题。
- 通俗:别人都在追 B站 热榜的时候,你已经在做**下一个**即将上 B站 热榜的题 —— 领先一个身位。

**A6. 选题矩阵 / 系列规划**
- 不只给单个视频,给一份"系统覆盖缺口"的系列计划,视频之间能互相导流。
- 通俗:从"今天做哪期" → "未来一个月这条赛道怎么打"。

**A7. 持续监控雷达(回应设计决策 #7)**
- 盯住一个赛道,缺口出现 / 对手出片 / 热度突变时**主动预警**(定时任务 + 通知)。
- 通俗:从"你主动来查" → "它替你盯着,有机会就喊你"。

> 进阶模块的可行性:A1/A3/A6 基本是纯计算,复用现有数据即可;A2 要字幕+弹幕(可能缺,需兜底);A4 要读你自己频道数据(用户空间 API 可读);A5 跨平台接口较杂(标 ⚠️ 实测);A7 要调度基建。**建议按能力阶梯渐进交付,不要一口吃成胖子**(见 §9)。

## 9. MVP(v0)—— 最小但能惊艳的 demo

**目标:** 单人几天,输入一个关键词,产出一份真实的选题雷达报告。

v0 范围(砍到最小):
1. 数据层:只做 `tool_search_topic` + `tool_get_video_hot_comments` 两个工具(够支撑 Step2/5)。
2. agent 流水线:Step1 查询扩展 → Step2 采集 → Step3 角度聚类 → Step5 缺口挖掘 → Step7 三方案。**先跳过** Step4 精细打分和 Step6 时机(用粗略版:数量+近 90 天占比)。
3. 输出:第 7 节那份 markdown 报告。
4. 跑通一个真实选题(比如就用 "MCP" 或 "Claude Code")出一份报告做 demo。

v0 明确**不做:** 字幕、跨平台、定时任务、写操作、Web UI。

**验收:** 给一个关键词,5 分钟内拿到一份"竞争态势+缺口+三方案+做不做"的报告,且评论缺口都有真实引用。

## 10. 风险与边界

- **反爬/限流:** 严格限速 + 缓存;demo 用缓存数据保证可复现。
- **接口漂移:** B站改版会破接口;采集层要把"字段解析"和"业务逻辑"解耦,坏一个字段不崩全局。
- **ToS / 隐私:** 只读公开数据;评论只做聚合分析 + 必要引用;不建用户画像。
- **不教标题党、不承诺数据:** 报告基调是"帮你判断要不要做、做哪个角度",不是"保你爆"。
- **诚实劝退:** 没机会的题就说放弃,这是 agent 的可信度来源。

## 11. 能力阶梯(分版本渐进交付)
